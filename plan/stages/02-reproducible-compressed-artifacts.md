# Phase 2 — Reproducible Compressed Artifacts

## Purpose

Turn the current compressed-artifact scaffolding into a reproducible packaged-data system for 1500-2500 CE. This phase depends on Phase 1 reference-quality source outputs because artifact fit errors must be measured against trustworthy inputs.

## Spec drivers

- `spec/requirements.md`: FR-8, FR-9, NFR-2, NFR-4
- `spec/data-compression.md`: segmented polynomial/residual format, stored-vs-derived outputs, generation pipeline
- `spec/backends.md`: `pleiades-data` responsibilities
- `spec/validation-and-testing.md`: artifact validation, benchmarks, release gates

## Current baseline

`pleiades-compression` defines an in-memory artifact model, deterministic binary codec, artifact capability profile metadata, lookup path, and explicit body/segment random-access helpers. `pleiades-data` exposes a packaged backend backed by a checked-in deterministic binary artifact fixture with an ecliptic-only/no-motion profile, and validation tooling can inspect, summarize, and regenerate that fixture from the checked-in JPL reference snapshot in tests.

   - Stamp source versions, generation parameters, and checksums.

The validation artifact summary and release summary now also render the packaged-artifact regeneration provenance line, and that provenance record now implements `Display`, so the maintainer-facing regeneration source stays visible in the compact release reports through the typed summary record instead of a duplicated string literal. The packaged-artifact regeneration summary now also has a typed `validate()` helper plus a report-facing wrapper that preflights the bundled body list against the checked-in reference snapshot before formatting the provenance line, which keeps the regeneration metadata and public-input coverage aligned at the reporting boundary. The packaged regeneration provenance now also records the artifact format version, checksum, and the current linear-segment generation policy, which makes the deterministic fit rule explicit alongside the reproducibility metadata. 3. Implement binary encode/decode and random access.
   - Move beyond sample in-memory construction to a stable serialized artifact representation.
   - Provide efficient lookup by body and time segment.
   - Validate boundary behavior around segment edges and unsupported bodies.
   - Keep artifact loading usable for desktop/server applications without mandatory native dependencies.
   - Progress note: the compression crate now prefers the later segment when two adjacent segments both include the same boundary instant, and the random-access regression suite covers that shared-boundary behavior alongside the existing missing-body and out-of-range checks.
   - Progress note: the compression codec now also rejects duplicate body entries, duplicate stored/derived/unsupported profile entries, and malformed segment metadata during encode/decode, which keeps the serialized artifact representation stricter and makes the random-access surface less ambiguous.

4. Measure compression quality.
   - Define initial per-body-class target error envelopes.
   - Compare decoded positions and speeds against generation sources.
   - Benchmark artifact size, lookup latency, batch throughput, and memory footprint.
   - Include measured limits in artifact metadata and validation reports.
   - Progress note: the artifact inspection report now also renders a packaged-artifact decode benchmark slice alongside the existing comparison envelope and boundary checks, so the compact validation path surfaces a concrete decode-cost measurement without waiting for a separate benchmark command. The same artifact inspection path now also surfaces an aggregate packaged-artifact boundary envelope across all bundled bodies, which makes segment-edge continuity easier to audit in the compact and detailed validation views. The packaged-artifact decode benchmark metadata now also validates its label, source, round, sample-count, and encoded-size fields before the report path formats it, which keeps the release-facing benchmark line from admitting malformed provenance or zero-sized coverage.

5. Upgrade `pleiades-data` to consume generated artifacts.
   - Bundle or fixture a small deterministic artifact for tests.
   - Provide feature-gated paths for larger distributable artifacts when appropriate.
   - Report packaged-data capabilities honestly through backend metadata and release profiles.
   - Progress note: the packaged backend now loads a checked-in binary artifact fixture and the tests compare it against a deterministic regeneration helper from the JPL reference snapshot, so the bundled artifact path is no longer a hand-authored segment chain. The compression crate now also exposes explicit body and segment random-access helpers, which makes the lookup path easier to reuse in future artifact validation and generation work. The compression artifact, body, and profile types now also expose public `validate()` helpers that mirror the codec's encode/decode checks, giving artifact generators a fail-fast path before serialization. The packaged-data regeneration path now validates the reconstructed artifact before encoding it, so fixture drift or generator regressions fail fast before the deterministic payload is written back to disk. The packaged body coverage summary now also validates against the bundled body set before report formatting, so packaged-body provenance drift now fails closed at the report boundary as well. The packaged-data backend now also exposes explicit artifact-from-bytes and optional artifact-from-path constructors, which gives the phase-2 loader a typed extension point for larger distributable artifacts without changing the default checked-in fixture path. The packaged mixed-frame and mixed TT/TDB batch-parity summaries now also validate their derived counts before release-facing formatting, which keeps the artifact transparency lines fail-closed if the frame or time-scale splits ever drift.

## Done criteria

- A maintainer can regenerate the test artifact from documented public inputs and deterministic parameters.
- `pleiades-compression` can encode, decode, checksum, and query the artifact format.
- `pleiades-data` uses generated artifacts rather than hand-authored sample segments for claimed coverage.
- Validation reports include artifact error summaries, boundary checks, and benchmark numbers.
- Artifact metadata states exactly what is stored, derived, or unsupported.
