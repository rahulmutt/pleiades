# v0.4.0 Release Cut Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship pleiades 0.4.0 — the first release-plz-automated release — by landing a pre-release toolchain-bump/doc-fix commit, merging the existing Release PR #7, and verifying the published crates.

**Architecture:** This is an operational release plan, not a feature build. Work happens directly on `main` (chore commits) plus the already-open release-plz Release PR #7 (`chore: release v0.4.0`, branch `release-plz-2026-07-04T07-10-33Z`). The release-plz GitHub Actions workflow (`.github/workflows/release-plz.yml`) does the publishing: merging PR #7 triggers the `release-plz release` job, which publishes all 16 publishable crates to crates.io in dependency order, tags `v0.4.0`, and creates the GitHub Release.

**Tech Stack:** Rust workspace (18 crates, 16 publishable), mise task runner, release-plz 0.3.160, gh CLI.

**Spec:** `docs/superpowers/specs/2026-07-16-v0.4.0-release-cut-design.md`

## Global Constraints

- The workspace audit requires `mise.toml`'s rust pin to **exactly equal** `Cargo.toml`'s `[workspace.package] rust-version` (rule `tool-manifest.rust-version-mismatch`); both must say `1.97.1` after Task 1.
- 16 publishable crates bump lockstep 0.3.0 → 0.4.0; `pleiades-cli` and `pleiades-validate` are `publish = false` and must never appear in the publish set.
- Chore commits go **directly to `main`** (repo's established pattern for toolchain bumps). Never commit to the release branch except to fix a wrong/misleading changelog entry.
- **Never hand-edit crate versions** — only release-plz changes them.
- Merging PR #7 is **irreversible** (publishes to crates.io). Ask the maintainer for explicit confirmation immediately before the merge, even in subagent-driven execution.
- Known-benign failure: every push to `main` *before* the release merge makes the `release-plz release` job fail on `pleiades-fict` tarball verification (unpublished 0.3.0 crate verifying against published `pleiades-types 0.3.0`). Do not chase this failure; it disappears once 0.4.0 publishes.
- Today's date for status lines: **2026-07-16**.

```rust
use pleiades_core::{CelestialBody, ChartEngine, EphemerisRequest, Instant, JulianDay, TimeScale};
use pleiades_data::PackagedDataBackend;

fn main() {
    let engine = ChartEngine::new(PackagedDataBackend::new());
    let request = EphemerisRequest::new(
        CelestialBody::Sun,
        Instant::new(JulianDay::from_days(2451545.0), TimeScale::Tdb),
    );
    let result = engine
        .position(&request)
        .expect("packaged Sun lookup should succeed");
    let ecliptic = result
        .ecliptic
        .expect("packaged Sun result should carry ecliptic coordinates");
    let lon = ecliptic.longitude.degrees();
    // Geometric solar ecliptic longitude at J2000.0 is ~280.37 deg.
    assert!(
        (279.0..=282.0).contains(&lon),
        "Sun longitude {lon} outside expected J2000 window"
    );
    println!("Sun @ JD 2451545.0 TDB: ecliptic longitude {lon:.6} deg — OK");
}
```
