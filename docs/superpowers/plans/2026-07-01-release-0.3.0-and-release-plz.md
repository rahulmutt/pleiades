# Pleiades 0.3.0 Release + release-plz Automation — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship pleiades 0.3.0 with a curated `CHANGELOG.md` (bumping 9 existing crates, first-publishing 4 new ones), then stand up release-plz + git-cliff + CI so 0.4.0 onward is a review-and-merge-a-PR flow.

**Architecture:** Two sequenced phases. **Part A** cuts 0.3.0 by hand using the existing `cargo-release` flow with a git-cliff-seeded, hand-curated changelog. **Parts B+C** land afterward: a shared `cliff.toml` changelog config, a `release-plz.toml` with unified `version_group` versioning, and a GitHub Actions workflow that opens release PRs and publishes on merge. The manual `release.toml` path is retained as a documented fallback.

**Tech Stack:** Rust workspace (16 crates), mise (tool manager), cargo-release 1.1.2, git-cliff, release-plz, GitHub Actions, crates.io.

## Global Constraints

- **Tool installation is mise-first, pinned.** Every new tool is added to `mise.toml` under `[tools]` as a `cargo:<tool>` entry with an exact pinned version — never a floating `latest` left in the file.
- **Unified versioning.** All 13 publishable crates stay lock-step on one version (`version.workspace = true` inheritance + release-plz `version_group = "pleiades"`). Never introduce per-crate divergence.
- **Changelog scope:** `feat`, `fix`, `perf`, and breaking changes only. `docs`, `test`, `chore`, `style`, `refactor` are hidden from the changelog.
- **Repo URL for commit links:** `https://github.com/rahulmutt/pleiades`.
- **13 publishable crates:** `pleiades-types`, `pleiades-apparent`, `pleiades-apsides`, `pleiades-time`, `pleiades-backend`, `pleiades-compression`, `pleiades-houses`, `pleiades-ayanamsa`, `pleiades-vsop87`, `pleiades-elp`, `pleiades-jpl`, `pleiades-eclipse`, `pleiades-core`. **3 non-publishable** (`publish = false`): `pleiades-cli`, `pleiades-data`, `pleiades-validate`.
- **The crates.io publish is irreversible and maintainer-gated.** No task fires `cargo publish` / `cargo release --execute` autonomously; the plan stops for the maintainer to run it.
- **CI parity:** release workflows reuse the mise-managed Rust toolchain (`jdx/mise-action@v2`) to match `.github/workflows/ci.yml`.

```toml
# Changelog config shared by the 0.3.0 bootstrap (git-cliff CLI) and all future
# releases (release-plz). Scope: feat / fix / perf / breaking only.
[changelog]
header = """
# Changelog

All notable changes to this project are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).
"""
body = """
{% if version %}\
## [{{ version | trim_start_matches(pat="v") }}] - {{ timestamp | date(format="%Y-%m-%d") }}
{% else %}\
## [Unreleased]
{% endif %}\
{% for group, commits in commits | group_by(attribute="group") %}
### {{ group | striptags | trim | upper_first }}
{% for commit in commits %}
- {{ commit.message | split(pat="\\n") | first | upper_first }} \
([{{ commit.id | truncate(length=7, end="") }}](https://github.com/rahulmutt/pleiades/commit/{{ commit.id }}))\
{% endfor %}
{% endfor %}
"""
trim = true

[git]
conventional_commits = true
filter_unconventional = true
protect_breaking_commits = true
filter_commits = true
tag_pattern = "v[0-9]*"
# `<!-- N -->` prefixes force section ordering; the template strips them via `striptags`.
commit_parsers = [
  { message = "^[a-z]+(\\(.+\\))?!:", group = "<!-- 0 -->Breaking Changes" },
  { body = ".*BREAKING CHANGE", group = "<!-- 0 -->Breaking Changes" },
  { message = "^feat", group = "<!-- 1 -->Added" },
  { message = "^fix", group = "<!-- 2 -->Fixed" },
  { message = "^perf", group = "<!-- 3 -->Performance" },
  { message = ".*", skip = true },
]
```

- [ ] **Step 4: Verify the config parses**

Run: `git-cliff --config cliff.toml --unreleased --context >/dev/null && echo OK`
Expected: prints `OK` with no template/parse errors.

- [ ] **Step 5: Commit**

```bash

git commit -m "chore(release): add git-cliff and shared changelog config"
```

### Task 6: Add the release-plz GitHub Actions workflow

Uses the official release-plz pattern: **one workflow file with two jobs** (release + release-pr) for correct concurrency handling. This is a minor refinement of the design's "two workflow files" — same PR/publish split, better concurrency control.

**Files:**
- Create: `.github/workflows/release-plz.yml`

**Interfaces:**
- Consumes: `release-plz.toml` (Task 5), `cliff.toml` (Task 1), repo secrets `CARGO_REGISTRY_TOKEN` and `RELEASE_PLZ_TOKEN` (documented in Task 7).
- Produces: on push to `main`, a "Release 0.4.0" PR (via `release-pr`) and, on merge of a version bump, a publish + tag + GitHub Release (via `release`).

- [ ] **Step 1: Create the workflow**

```yaml
name: release-plz

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

jobs:
  # Publishes crates and creates tags/releases when a version bump lands on main.
  release-plz-release:
    name: release-plz release
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Set up mise
        uses: jdx/mise-action@v2
        with:
          install: false
      - name: Install Rust toolchain
        run: mise install rust
      - name: Run release-plz release
        uses: release-plz/action@v0.5
        with:
          command: release
        env:
          GITHUB_TOKEN: ${{ secrets.RELEASE_PLZ_TOKEN }}
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}

  # Opens/updates the release PR that bumps versions and prepends the changelog.
  release-plz-pr:
    name: release-plz PR
    runs-on: ubuntu-latest
    concurrency:
      group: release-plz-pr
      cancel-in-progress: false
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Set up mise
        uses: jdx/mise-action@v2
        with:
          install: false
      - name: Install Rust toolchain
        run: mise install rust
      - name: Run release-plz release-pr
        uses: release-plz/action@v0.5
        with:
          command: release-pr
        env:
          GITHUB_TOKEN: ${{ secrets.RELEASE_PLZ_TOKEN }}
```

- [ ] **Step 2: Verify the workflow is valid YAML**

Run: `python3 -c "import yaml,sys; yaml.safe_load(open('.github/workflows/release-plz.yml')); print('OK')"`
Expected: prints `OK`.

- [ ] **Step 3: Commit**

```bash

git commit -m "ci(release): add release-plz workflow"
```

```markdown
## Releasing

Releases are automated with [release-plz](https://release-plz.dev). On every
push to `main`, release-plz maintains a **Release** pull request that bumps all
crates to the next unified version and updates `CHANGELOG.md` from Conventional

publish all publishable crates to crates.io, and create the GitHub Release.

### Required repository secrets

  publish scope for the `pleiades-*` crates.
- `RELEASE_PLZ_TOKEN` — a GitHub token used by the workflow so its PRs and tags
  trigger CI. Prefer a **GitHub App** installation token (scoped, rotating);
  a fine-grained PAT with `contents: write` + `pull-requests: write` also works.
  The default `GITHUB_TOKEN` cannot trigger downstream workflows, so it is not
  sufficient here.

### Manual fallback

To cut a release by hand (e.g. if crates.io automation is unavailable), use the
retained `release.toml` config: `cargo release <version> --execute`.
```
