# Pleiades 0.3.0 Release + release-plz Automation — Design

**Date:** 2026-07-01
**Status:** Approved (design)

## Summary

Two related pieces of work:

1. **Ship 0.3.0 now** with a proper, curated `CHANGELOG.md`. This bumps the 9
   crates already on crates.io and first-publishes 4 new crates that were added
   during the 556 commits since v0.2.0.
2. **Stand up release automation** (release-plz + git-cliff + CI) so that 0.4.0
   and every release afterward is a review-and-merge-a-PR flow, while preserving
   the workspace's existing unified (lock-step) versioning.

The two pieces are **sequenced, not simultaneous**: 0.3.0 ships first by hand
(Part A), then the automation lands (Parts B + C) so its first automated target
is a clean 0.3.0 → 0.4.0 transition rather than a messy 556-commit backfill.

## Context (current state)

- Rust workspace, 16 crates. `version.workspace = true` inheritance — all crates
  share one version field (`[workspace.package] version`), currently `0.2.0`.
- **13 publishable crates**, **3 marked `publish = false`** (`pleiades-cli`,
  `pleiades-data`, `pleiades-validate`).
- Publishable crates split by crates.io state:
  - **On crates.io at 0.2.0 (9):** `pleiades-ayanamsa`, `pleiades-backend`,
    `pleiades-compression`, `pleiades-core`, `pleiades-elp`, `pleiades-houses`,
    `pleiades-jpl`, `pleiades-types`, `pleiades-vsop87`.
  - **Never published (4):** `pleiades-apparent`, `pleiades-apsides`,
    `pleiades-eclipse`, `pleiades-time`. These first-publish at 0.3.0.
- Releases v0.1.0 / v0.2.0 were cut with **`cargo-release`** via `release.toml`
  (`shared-version = true`, `publish = true`, `push = true`, `tag = true`).
- Commits already follow **Conventional Commits**. 556 commits since v0.2.0:
  ~200 `feat`, 58 `fix`, plus docs/test/chore/etc. Breaking changes are marked
  (`feat(houses)!`, `feat(ayanamsa)!`).
- **No `CHANGELOG.md` exists yet.**
- CI (`.github/workflows/ci.yml`) is minimal: `mise run ci`. No release
  automation in Actions today.

## Decisions (locked)

| Decision | Choice |
|---|---|
| Future automation tool | **release-plz** (Rust-native; built on cargo-release + git-cliff) |
| 0.3.0 changelog method | **Auto-generate with git-cliff, then hand-curate** |
| Changelog scope (ongoing) | **feat, fix, perf + breaking**; hide docs/test/chore/style/refactor |
| Versioning model | **Unified / shared** — all crates bump lock-step |
| How 0.3.0 is cut | **Bootstrap by hand** with existing cargo-release, then automate |

## Part A — Bootstrap 0.3.0 by hand (this release only)

Independent of Parts B/C. Steps:

1. **Generate raw changelog** as a starting point:
   `git-cliff v0.2.0..HEAD`.
2. **Curate into `CHANGELOG.md`** (Keep-a-Changelog format), `0.3.0` section:
   - Lead with **Highlights**: the 4 new crates (`apparent`, `apsides`,
     `eclipse`, `time`) and the breaking changes.
   - Sections: `### Breaking Changes`, `### Added` (feat), `### Fixed` (fix),
     `### Performance` (perf).
   - Drop docs / test / chore / style / refactor noise.
3. **Cut the release with the existing `cargo-release`** flow (`release.toml`
   already handles shared-version bump, `v0.3.0` tag, push, `cargo publish`).
   The 4 new crates first-publish automatically (they are `publish`-enabled and
   inherit the workspace version).
4. **Verify**: all 13 crates on crates.io at 0.3.0; `v0.3.0` tag + GitHub
   Release exist; `CHANGELOG.md` renders correctly.

## Part B — git-cliff config (`cliff.toml`, repo root)

Single source of truth for changelog formatting, used by both the Part A
bootstrap and all future release-plz runs (so 0.3.0 and 0.4.0+ look identical).

- Commit groups, in order: **Breaking → Added (`feat`) → Fixed (`fix`) →
  Performance (`perf`)**. All other types skipped.
- Parse `!` and `BREAKING CHANGE:` into the Breaking section.
- Strip Conventional-Commit scopes into readable prose.
- Link each entry to its commit/PR on the repo.

## Part C — release-plz + CI automation

### 1. `release-plz.toml` (repo root)

- `[workspace]`:
  - `changelog_config = "cliff.toml"` — reuse the Part B config.
  - `git_release_enable = true` — create GitHub Releases.
  - `dependencies_update = false` — don't churn `Cargo.lock` on release.
- Per-crate `[[package]]` entries assigning all **13 publishable crates** to a
  single `version_group = "pleiades"` → lock-step unified version bumps
  (0.3.0 → 0.4.0). The 3 `publish = false` crates are auto-skipped.
- **Known nuance:** release-plz only *publishes* crates that actually changed,
  but the shared version number stays uniform via `version.workspace = true`
  inheritance. For this workspace (nearly everything changes each cycle) this is
  acceptable; an untouched crate simply skips a version number on crates.io.

### 2. GitHub Actions workflows (`.github/workflows/`)

- **`release-plz-pr.yml`** — on push to `main`: runs `release-plz release-pr`.
  Opens/updates a "Release 0.4.0" PR that bumps versions and prepends the
  curated changelog. Reviewer merges when ready.
- **`release-plz-release.yml`** — on push to `main`: runs `release-plz release`.
  On a merged version bump, it tags, `cargo publish`es changed crates in
  dependency (topological) order, and creates the GitHub Release.
- Both use the official release-plz action on `ubuntu-latest`, reusing the
  `mise`-managed Rust toolchain for consistency with `ci.yml`.

### 3. Secrets & prerequisites (repo settings; documented for the maintainer)

- `CARGO_REGISTRY_TOKEN` — crates.io token for publishing.
- A **PAT or GitHub App token** for the release-plz PR job, so its PR/pushes
  trigger downstream CI (the default `GITHUB_TOKEN` cannot). GitHub App is the
  more secure option and will be noted as preferred.

### 4. Retire `release.toml`

- Keep it as a **documented manual fallback** (the Part A path stays valid for
  emergencies); release-plz becomes the default.
- Add a short "Releasing" section to `AGENTS.md` / `README.md` pointing at the
  new flow.

## Verification

- **Part A:** 13 crates visible on crates.io at 0.3.0; `v0.3.0` tag + GitHub
  Release exist; `CHANGELOG.md` renders correctly.
- **Part C:** run the release-plz action in `--dry-run` first; confirm it opens
  a sane release PR against a test commit before trusting the publish job with
  the crates.io token.

## Out of scope

- Migrating to independent per-crate versioning (explicitly rejected).
- Changing the crates.io distribution target or the set of publishable crates.
- Reworking `ci.yml` beyond what the release workflows need.
