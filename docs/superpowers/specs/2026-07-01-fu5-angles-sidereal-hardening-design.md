# FU-5 angles & sidereal-time hardening — design

**Date:** 2026-07-01
**Origin:** `docs/follow-ups.md` § FU-5 (SP-1 angles & sidereal-time deferred items),
opened by the `feat/sp1-angles-sidereal` whole-branch review.
**Scope:** the three open FU-5 items. The two "open by design" items elsewhere in
`follow-ups.md` (SE Moon Moshier ceiling, ELP of-date equatorial) are intentionally
untouched.

A small three-part sub-project: one focused single-source refactor plus two
test-coverage additions. **No behavior change is intended anywhere** — every part
is locked by an existing or new gate.

---

## Item 1 — GMST / equation-of-equinoxes single-source

### Current state (verified)

- `pleiades-apparent::sidereal::greenwich_mean_sidereal_time_degrees` and
  `pleiades-time::sidereal::gmst_degrees` are the **same polynomial**. The only
  difference: apparent's returns the value **un-normalized** (its caller
  `sidereal_time` re-normalizes `gmst + ee + lon` once at the end); time's returns
  `[0, 360)`.
- The equation-of-equinoxes formula `Δψ · cos(ε_true)` is implemented in
  `pleiades-apparent/src/sidereal.rs::equation_of_equinoxes_degrees` **and**
  hand-inlined at `crates/pleiades-core/src/chart/mod.rs:411`.
- Dependency graph: `pleiades-apparent → pleiades-types` only;
  `pleiades-time → pleiades-types` only. Adding **`pleiades-apparent → pleiades-time`
  is acyclic** and is a valid consolidation direction. (`pleiades-core` already
  depends on both and already calls `pleiades_time::gmst_degrees` in its
  topocentric-LAST path.)

### Plan

1. Add `pleiades-time` as a dependency of `pleiades-apparent`. Reimplement
   `greenwich_mean_sidereal_time_degrees` as a thin wrapper delegating to
   `pleiades_time::gmst_degrees`. Because `sidereal_time` re-normalizes
   `gmst + ee + lon` at the end (and normalizes each field via
   `Angle::normalized_0_360`), delegating to the **normalized** variant is
   numerically identical to the prior un-normalized path. This equivalence is
   asserted (see Testing).
2. Single-source the equation of equinoxes. Extract one helper — e.g.
   `equation_of_equinoxes(delta_psi_deg: f64, true_obliquity_deg: f64) -> f64`
   computing `delta_psi_deg * true_obliquity_deg.to_radians().cos()` — that both
   `apparent/sidereal.rs::equation_of_equinoxes_degrees` and the `chart/mod.rs:411`
   site call. The chart site keeps its already-computed `nut` / `true_obliquity`
   and passes them in (no extra nutation table lookup); the `cos(ε)` formula then
   has exactly one home.
3. **Lock-in:** add a cross-crate agreement test over a JD sweep asserting
   apparent's GMST equals time's GMST (modulo normalization), so a future
   coefficient edit to one crate cannot silently diverge from the other.

### Explicitly out of scope (stays for SP-2)

Migrating the `chart/mod.rs` topocentric-LAST path onto the public `sidereal_time`
API. That path converts **TT → UT1** first, whereas the public `sidereal_time`
consumes the `Instant` JD **as supplied** (UT1 policy / Global Constraint). Merging
them touches the time-scale contract and is earmarked for SP-2 in FU-5. This
sub-project only removes the duplicated **formula**, not the differing time-scale
handling.

## Item 2 — Southern-hemisphere `validate-angles` gate row

### Current state (verified)

- The `validate-angles` corpus (`crates/pleiades-validate/data/houses-corpus/angles.csv`)
  is **northern-only**: lat 0 / 40 / 55 / 66 (plus a second lat-40 epoch). So
  `asc_mc_from`'s `f_pole = -90 - lat` pole-height branch and the vertex
  western-hemisphere flip (`crates/pleiades-houses/src/systems/mod.rs`, ~lines
  186–207) are exact `swehouse.c` ports with **no gate row**.
- The corpus is generated by `tools/se-house-reference` from a hardcoded `fixtures`
  list (`main.rs:76–80`). That single list drives **all three** corpus slices:
  `cusps.csv` (115 rows), `sectors.csv` (5 rows), `angles.csv` (5 rows).
- `libclang-19` is present at `/lib/x86_64-linux-gnu/libclang-19.so`; building the
  tool only needs `LIBCLANG_PATH=/lib/x86_64-linux-gnu` set.

### Plan (approach A — regenerate all slices)

1. Add one southern fixture at **lat −33°** (representative southern latitude, not
   an edge case) to the `se-house-reference` `fixtures` list. Verify the generated
   row actually drives the intended branch (`f_pole = -90 - lat`); if the vertex
   western-hemisphere flip does not trigger at that geometry, adjust the fixture's
   longitude/epoch to force it so the row earns its place.
2. Rebuild `se-house-reference` (with `LIBCLANG_PATH` set) and regenerate **all
   three** slices — the southern fixture naturally extends `cusps.csv`,
   `sectors.csv`, and `angles.csv`. Update `manifest.txt` row counts and all three
   slice checksums.
3. Confirm `validate-angles` and the houses cusp/sectors gates pass on the new row
   within their existing ceilings.

**Risk / discipline:** a southern cusp row could surface a latent
southern-hemisphere discrepancy in some exotic house system. If it does, it is
**surfaced as a finding** (new follow-up), not silently patched by loosening a
ceiling.

## Item 3 — High-latitude Porphyry-fallback `asc_mc` consistency test

### Current state (verified)

`asc_mc` is constructed at two sites in `crates/pleiades-houses/src/systems/mod.rs`:

- The **main** end-of-function `HouseSnapshot` (~line 497) — covered by the existing
  `asc_mc` consistency test.
- The **early-return Porphyry high-latitude fallback** `HouseSnapshot` (~line 404) —
  structurally identical, verified by inspection only, with **no assertion**.

### Plan

Add one unit test that forces the fallback branch: a Porphyry-fallback-eligible
system (e.g. Placidus or Koch) at a latitude beyond its normal bound
(|lat| > ~66°) so construction takes the ~line-404 early-return path. Assert that
`snapshot.asc_mc` equals an independent
`asc_mc_from(local_sidereal_time(instant, lon).degrees(), lat, obliquity.degrees())`
computation. Pure unit test, no corpus.

## Testing & ordering

- **Order:** Item 1 (refactor) → Item 3 (unit test) → Item 2 (corpus regen). Item 1
  first because it is the only code-behavior change; the agreement test plus the
  existing `validate-angles` (~0.16″ ceiling) and houses gates confirm no numeric
  drift. Items 2 and 3 are mutually independent.
- **TDD:** Items 1 and 3 are test-first (the GMST agreement test and the fallback
  consistency test are written before/with the change). Item 2 is
  corpus-generation + checksum, verified by the gate passing on the new row.
- **Gates that must stay green:** `validate-angles`, the houses cusp/sectors gates
  (Item 2 touches their corpus), plus workspace-wide `cargo test` / `cargo fmt` /
  `cargo clippy`.
- **Build-env note (record for FU/CLAUDE):** `tools/se-house-reference` requires
  `LIBCLANG_PATH=/lib/x86_64-linux-gnu` to build. This is **not** required to run
  any gate or build the workspace — the gates read the committed CSVs via
  `include_str!` and never rebuild the tool.

## Risks

- **Item 1:** the normalized-vs-un-normalized GMST swap. Mitigated by the
  cross-crate agreement test and the existing `validate-angles` gate — any drift
  above ~0.16″ would fail the gate.
- **Item 2:** a southern cusp row could expose a latent southern-hemisphere bug →
  surfaced as a finding, not silently patched.
- **Item 3:** none material; additive test only.

## Success criteria

- No duplicated GMST polynomial and no duplicated equation-of-equinoxes formula;
  both single-sourced, with a cross-crate agreement test guarding GMST.
- `validate-angles` corpus contains at least one southern-latitude row exercising
  the `f_pole = -90 - lat` branch (ideally the vertex flip too), with regenerated
  manifest/checksums, all gates green.
- The Porphyry high-latitude fallback `asc_mc` site is covered by an assertion.
- FU-5's three open items are closed (or explicitly re-deferred if Item 2 surfaces
  a discrepancy warranting its own follow-up).
